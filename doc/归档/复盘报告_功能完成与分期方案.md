> 基线：产品设计说明书 V3.0、Agent 施工方案 V2.0、ZCODE.md 项目宪法，对照实际代码仓库（commit `7b931a5`）。

## 0. 关键不一致预警（必须先认知）

施工方案 §1.1 和 ZCODE.md 描述的目标技术栈是 **monorepo（apps/web + apps/api + packages/）、Cesium、Fastify、Prisma+PostGIS、BullMQ+Redis、react-router v6、Zustand+TanStack Query**；而**实际代码是一个单仓 Vite 应用**：

| 维度 | 文档假设 | 代码实际 | 差距 |
|---|---|---|---|
| 仓库结构 | pnpm workspaces monorepo | 单仓 `family-atlas-demo`（npm，package-lock） | 重构级 |
| 路由 | react-router v6 多路由 | 无路由，单文件 App + 锚点滚动 (`main.jsx:542,546`) | 缺失 |
| 状态 | Zustand + TanStack Query | 纯 `useState` + 散落 `localStorage` | 缺失 |
| 3D 地球 | Cesium + deck.gl | **three.js**（`cobe-globe-weather.tsx` 实为 raw three.js） | 栈不同 |
| 2D 降级 | MapLibre GL | **Leaflet** + Esri 瓦片 | 栈不同 |
| 后端 | Fastify + Prisma + PostGIS | 仅 1 个 Cloudflare Pages Function (`functions/api/scrape-proxy.ts`) | 缺失 |
| 队列 | BullMQ + Redis | 无 | 缺失 |
| 测试 | Vitest + Playwright + supertest | 无 test 脚本、无测试文件 | 缺失 |

**含义**：规格定义的后端、数据持久化、队列、SSE、Provider Adapter 体系**全部尚未开工**。施工方案的"保留现有栈、禁止整仓重写"原则在**前端可视化层**成立（three.js/Leaflet 可继续用），但在**后端/数据层**需要从零搭建。

## 核心结论

1. **当前工程实质处于"V0.6 可视化原型"**--地球/地图/景点 POI 管道扎实可用，但规格的六大业务对象中 Journey/Memory/Share/Home 四个完全缺失，后端与数据持久化层零。
2. **存在一处必须先决策的根本冲突**：规格假设 monorepo+Cesium+Fastify+PostGIS+BullMQ，实际是单仓 Vite+three.js+Leaflet+Cloudflare Pages。前端栈可保留，但后端/数据层需决定走"贴合规格补齐重型栈"还是"轻量 Cloudflare 全家桶"。
