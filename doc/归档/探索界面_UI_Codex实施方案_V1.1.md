> 本文档是可直接交给 Codex 的施工输入。  
> 目标：在现有项目内完成探索界面全部冻结改动，不推翻既有品牌资产，不创建与现有技术栈平行的新系统。

## 0. Codex 执行指令

### 0.1 决策优先级

遇到冲突时按以下顺序执行：

1. 用户已确认决策；
2. 项目现有《产品设计说明书》；
3. 本文档；
4. GPT 最终冲突裁决；
5. Claude UI 工程骨架；
6. DeepSeek 局部交互参考；
7. 现有代码中的临时占位实现。

不得使用千问、Grok 或历史草案中的冲突条目覆盖本文档。

### 0.2 开工前必须完成

先扫描仓库，不要立即重写：

1. 找出探索首页、洲页、国家页、地区页、POI 详情页的路由和组件。
2. 找出当前地图引擎、3D 地球引擎、状态管理、API 层、样式系统、图标库和测试框架。
3. 找出 Journey、Wishlist、PrepPlan、Visited、Region Taxonomy、Weather 数据对应的模型和接口。
4. 建立一份影响清单，列出将修改、新增、删除的文件。
5. 优先复用现有组件、请求层和数据模型；除非现有实现无法满足需求，不得替换地图或状态管理框架。
6. 不得伪造后端数据。后端能力缺失时，建立明确的 adapter/interface、loading、empty、unavailable 状态，并在代码中留下可搜索的 TODO。
7. 所有新增功能必须有测试；所有旧功能必须通过回归测试。

## 0.4 Gemini 新增意见裁决

### 采纳

1. **临近出发时间感知**：纳入既有 `departure_soon` 首页模式。默认触发窗口为出发前 72 小时，但必须通过配置读取，不得写死。
2. **目的地陈旧数据自动刷新**：进入目的地时，如本地数据超过新鲜度阈值，可后台自动刷新。
3. **AI 语义筛选**：列为 P2 Feature Flag 功能，通过 Provider Adapter 将自然语言转为现有筛选条件。
4. **视觉降噪与移动 Bottom Sheet**：与原冻结规范一致，作为补充确认。
5. **轻量 Spring Motion**：只用于卡片、点位、抽屉的微交互，建立统一 Motion Token。

### 部分采纳或修正

1. **72 小时**：作为默认值，不作为业务硬编码。不同 Journey 类型、时区和交通方式可配置不同阈值。
2. **静默同步**：不能替代用户已确认的手动同步、更新时间和进度反馈。自动刷新仅是附加能力。
3. **阻尼系数 `ζ=0.75`**：不作为跨技术栈的固定验收值。目标手感设为轻微回弹、快速收敛；由当前动画库调参，推荐阻尼比约 `0.70–0.85`。
4. **工具卡 85% 透明度**：不固定为单一数值。通过 Token 在约 `82%–88%` 范围内适配背景，并以文字对比度 `≥4.5:1` 为验收标准。
5. **AI 语义筛选自动应用**：不得无提示直接改动结果。必须先展示解析出的筛选胶囊，允许用户编辑或确认。

### 拒绝

1. **将 Spring Physics 应用于地图拖拽和相机主导航**：地图引擎自身已有惯性机制，叠加自定义弹簧可能造成晕动、失控和输入延迟。
2. **将 AI 语义筛选设为 P1 或首发阻塞项**：依赖模型服务、成本、延迟与隐私策略；必须作为 P2 增强并保留完全可用的手动筛选。
3. **只有静默同步、不展示同步入口**：违反已确认的可信数据与可见进度要求。

## 2. 建议的数据模型

以下为 TypeScript 形态示例；若项目使用其他语言，建立等价结构。

```ts
type HomeMode =
  | "new_user"
  | "explore"

  | "departure_soon";

type DestinationLifecycle =
  | "untouched"
  | "wishlist"
  | "preparing"
  | "ready"
  | "in_journey"
  | "completed"
  | "visited_unlogged";

type DataFreshnessStatus =
  | "fresh"
  | "usable"
  | "stale"
  | "unavailable"
  | "syncing";

interface RegionTaxonomy {
  id: string;

}

interface FitnessBreakdown {
  seasonWeather?: number;
  familyFriendly?: number;
  transport?: number;
  crowding?: number;
  openness?: number;
  risk?: number;
}

interface FitnessResult {
  score: number | null;
  breakdown: FitnessBreakdown;
  calculatedAt: string | null;
  confidence: "high" | "medium" | "low" | null;
  isDynamic: boolean;
  isExplainable: boolean;
}

interface DataLayerFreshness {
  layer: "daylight" | "cloud" | "precipitation" | "wind" | "severe_weather";
  updatedAt: string | null;
  status: DataFreshnessStatus;
  sourceLabel?: string;
}

interface DestinationLiveStatus {
  weatherUpdatedAt?: string;
  openingUpdatedAt?: string;
  trafficUpdatedAt?: string;
  eventUpdatedAt?: string;

  sourceLabels: string[];
}

interface SelectedMapEntity {
  id: string;
  type: "continent" | "country" | "region" | "poi";

```

### 2.1 术语映射

- `点亮`：只用于地图和叙事，事实为已到访；
- `已到访`：数据统计表达；
- `心愿`：收藏待规划；
- `准备中`：PrepPlan 未完成；
- `解锁`：必要准备完成；
- `Journey`：进入正式旅程；
- `适配度`：计算结果，不是生命周期状态；
- `最近浏览`：行为记录，不计入进度；
- 禁止新增 `发现` 状态；
- 禁止用 `locked` 表示地区不可浏览。

## 3. 设计 Token 与组件基线

### 3.1 状态色

若项目已有 Token，扩展现有 Token，不要在组件中散写色值。

```css
:root {
  --atlas-accent-selected: #E8833B;
  --atlas-state-live: #4FC3F7;
  --atlas-state-normal: #66BB6A;
  --atlas-state-opportunity: #FFCA28;
  --atlas-state-danger: #EF5350;
  --atlas-state-muted: #9E9E9E;
  --atlas-card-archive: #F5F0E8;
  --atlas-card-tool: rgba(6, 18, 17, 0.84); /* 可在 0.82–0.88 内按背景调整 */
}
```

颜色只表达状态和动作。类别由 SVG 图标与文字表达。

### 3.2 卡片组件

仅保留：

```tsx
<ArchiveCard /> // 米白，叙事、区域背景、准备信息
<ToolCard />    // 深色半透明，筛选、地图状态、同步、操作；背景不固定 85%，通过 Token 调整
```

禁止新增第三种视觉语法。橙色错位厚阴影只允许 `ArchiveCard` 使用。

### 3.4 Motion Token

不要在各组件内分别写动画参数。建立统一 Motion Token：

```ts
interface AtlasMotionToken {
  durationMs?: number;
  stiffness?: number;
  damping?: number;
  mass?: number;
}

const atlasMotion = {
  markerHover: { durationMs: 180 },
  panelOpen: { durationMs: 280 },
  gentleSpring: {
    // 目标阻尼比约 0.70–0.85；按当前动画库换算，不以 ζ=0.75 作为硬编码验收
    stiffness: 220,
    damping: 24,
    mass: 1,
  },
};
```

`gentleSpring` 仅允许用于：

- 档案卡轻微吸附；
- 地图点位选中回弹；
- Bottom Sheet 停靠；
- 卡组切换完成后的轻微收敛。

禁止用于：

- 地图相机平移和缩放主轨迹；
- 长距离路由转场；
- 同步进度；
- 风险提示。

开启 `prefers-reduced-motion` 或产品“减弱动态”后，Spring Motion 必须替换为无回弹短淡入。

### 3.3 基础尺寸

- 正文：最小 14px；
- 辅助文本：最小 12px；
- 面板标题：24–30px；
- 可点击高度：最小 36px；
- 暗底关键文字对比度：至少 4.5:1；
- 地图视口安全区：8px；
- 地区/POI 右面板桌面最小宽度：360px。

# 4. P0 详细实施

## UI-P0-01：首页状态分流

### 实施

建立统一的 `resolveHomeMode()`：

```ts
function resolveHomeMode(ctx: {
  isFirstVisit: boolean;
  activeJourney?: Journey;
  departureThresholdHours: number; // 默认 72，由配置或实验参数提供
}): HomeMode
```

判定顺序：

1. 存在未完成 Journey 且距离出发时间在 `departureThresholdHours` 内 → `departure_soon`；默认值 72 小时，但不得硬编码；
2. 存在未完成 Journey → `active_journey`；
3. 首次访问 → `new_user`；
4. 其他 → `explore`。

不要复制四套页面。使用同一首页 Shell，根据 `HomeMode` 切换主内容。

### `departure_soon` 具体展示

- 地球定位出发地与目的地；
- 显示路线弧线和目的地气象状态；
- 主卡切换为“出发检查”；
- 展示证件、交通、住宿、天气、风险、关键预约的完成状态；
- 风险信息必须来自目的地同步结果；
- 高风险或同步失败不能只用颜色表达，必须有文字原因；
- 用户时区、目的地时区和出发时间必须统一换算后再判断阈值。

### 验收

- 四种模式可用 fixture 或测试账号稳定复现；
- 刷新页面后模式不闪烁；
- Journey 请求 loading 时显示稳定骨架，不先闪完整探索首页。

## UI-P0-02：Active Journey 首页

### 实施

有未完成 Journey 时：

- Journey 卡为交互主角；
- 地球仍占背景主舞台；
- 自动定位出发地和目的地；
- 绘制路线弧线；
- 显示旅程名称、出发日期、准备度、待确认事项数；
- 主 CTA 为“继续准备”或“继续规划”，由生命周期决定；
- 提供次级入口“继续探索世界”。

不得把地球完全移除，也不得继续显示无关的三张本季卡抢占主视觉。

### 验收

- Journey 卡信息来自真实 Journey/PrepPlan；
- 点击路线或“查看路线”进入对应 Journey；
- 地球定位失败时 Journey 卡仍可使用。

## UI-P0-03：Explore 首页

### 实施

无未完成 Journey 时：

- 动态 3D 气象地球为第一视觉；
- 新用户显示轻量引导：“从哪里开始探索？”；
- 非新用户显示探索档案和本季卡组；

“从亚洲开始”应优先定位用户所在洲；没有定位或身份信息时使用产品默认洲，并在文案中避免误导。

### 验收

- 页面首屏只有一个明确主视觉；
- 新用户引导可关闭且不重复干扰；
- 随机发现不会返回不可访问或数据失效的目的地。

## UI-P0-04：LIVE ATMOSPHERE

### 实施

将绝对实时表达替换为：

- 主标识：`LIVE ATMOSPHERE`；
- 中文：`全球旅行环境演算`；
- 展示昼夜、云层、降水、风场的更新时间；
- 昼夜标记为“实时天文演算”；
- 气象层展示真实 `updatedAt`。

建立 `DataFreshnessBadge`，按数据状态呈现：

- fresh：青色；
- usable：米白；
- stale：黄色并提示“数据可能已变化”；
- unavailable：灰色并关闭对应层；
- syncing：进度状态。

### 验收

- 不出现“全球全部数据实时”之类文案；
- 每个气象层有单独新鲜度；
- 数据失效时不继续播放伪实时动画。

## UI-P0-05～07：两级同步系统

### 全球同步

创建 `GlobalAtmosphereSyncPanel`：

- 云层；
- 降水；
- 风场；
- 显著天气。

### 目的地同步

创建 `DestinationLiveSyncPanel`：

- 当地天气；
- 景区开放；
- 交通；
- 当地事件；
- 风险提醒。

两者使用不同 API/缓存键/更新时间，不得通过一个笼统刷新按钮同时假装完成。

### 进入目的地时的后台刷新

- 读取每类数据的 freshness threshold；
- 未过期：直接使用缓存，不重复请求；
- 过期：后台自动刷新，但继续展示上一次成功数据；
- 严重过期或无缓存：展示 skeleton/unavailable 状态；
- 自动刷新期间仍显示紧凑的同步状态；
- 用户始终可以点击“同步最新信息”查看分项进度或重试；
- 自动刷新失败不得清空上一次成功数据。

### 状态

每项支持：

- idle；
- queued；
- syncing；
- success；
- partial；
- failed；
- unavailable。

允许部分成功。同步失败必须保留上一次成功数据并明确“最后成功更新于”。

### 验收

- 两级同步互不污染；
- 分项进度真实对应请求状态；
- 失败可以单项重试；
- 目的地详情展示来源标签，如“以当地官方公告为准”。

## UI-P0-15：CTA 状态矩阵

创建唯一的 CTA resolver：

- 操作失败时回滚乐观更新并提示原因。

## UI-P0-16～18：适配度系统

### 显示门槛

只有同时满足以下条件才显示数字：

```ts
const canShowFitness =
  result.score !== null &&
  result.isDynamic &&
  result.isExplainable &&
  Boolean(result.calculatedAt);
```

否则显示：

- “适配度计算中”；
- 或“当前适配度暂不可用”。

### 展开内容

最多显示 3–5 项：

- 季节与天气；
- 家庭友好；
- 交通便利；
- 拥挤程度；
- 开放/风险。

显示计算时间和置信度。

### 排序

- 有有效分数：按适配度降序；
- 并列：按置信度、数据新鲜度、稳定名称顺序；
- 无分数：放在组内末尾；
- 不允许使用随机数拉开分差。

### 数据质量

允许真实并列。只在以下情况报警：

- 大量对象总分和全部分项完全相同；
- 静态固定值；
- 更新时间完全相同且长期不变；
- 显示分数但没有构成数据。

### 验收

- 无假 86；
- 展开信息和总分来自同一结果；
- 页面换用户、季节或开放状态后可重新计算。

## UI-P2-11：AI 语义筛选

### 产品行为

在现有手动筛选上方增加可选输入框：

```text
描述你想去的地方
例如：适合带 3 岁孩子、7 月看花海、交通不要太折腾
```

AI 不直接替代筛选系统，而是把自然语言解析成现有结构化条件，例如：

```json
{
  "experiences": ["flower", "family"],
  "travelers": ["toddler"],
  "month": 7,
  "transportDifficultyMax": "low",
  "stayDuration": "one_day"
}
```

解析完成后先展示“已理解条件”胶囊：

```text
亲子 · 3岁幼儿
7月
花海
交通难度较低
[应用条件] [修改]
```

用户确认后才应用筛选和地图高亮。

### 技术结构

```ts
interface SemanticFilterProvider {
  parse(
    query: string,
    context: {
      locale: string;
      currentRegionId?: string;
      availableFilterSchema: FilterSchema;
    }
  ): Promise<SemanticFilterParseResult>;
}

interface SemanticFilterParseResult {
  filters: DestinationFilters;
  confidence: number;
  unsupportedTerms: string[];
  explanation?: string;
}
```

实现要求：

- 通过 `semanticFilterEnabled` Feature Flag 控制；
- Provider 与 UI 解耦，可切换远程模型、内部服务或规则解析器；
- 不得由前端伪造 AI 结果；
- 无服务、超时、低置信度时保留输入内容并引导使用手动筛选；
- 不记录敏感自由文本，除非隐私策略明确允许；
- 解析出的条件必须符合现有 Filter Schema；
- 不支持的条件明确标记，不偷偷忽略；
- 结果必须可编辑、可撤销；
- 语义筛选和手动筛选使用同一 URL/state 表达，避免两套逻辑。

### 验收

- Feature Flag 关闭时不影响现有筛选；
- 高置信度结果可正确转为筛选胶囊；
- 低置信度不会自动应用；
- 用户可逐项删除解析条件；
- 服务失败时手动筛选仍完整可用；
- 地图高亮与列表结果使用同一结构化筛选状态。
